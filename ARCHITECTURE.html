Section 5: Bottleneck Ranking

Rank Component Time Type Fix Direction
1 Extract LLM ~22s Compute-bound (output generation) Shrink output schema further; already done entity-type skip; Haiku max output is ~2,400 tokens — near floor
2 Judge LLM ~14s Compute-bound (Sonnet TTFT + generation) Merge judge+act into single Sonnet call (saves ~4s), or accept as Sonnet floor
3 Validate LLM ~13s Compute-bound (Sonnet) Most controversial: costs 13s, adds stance/triggers/peer_context. Worth it only if validate catches real errors
4 sec_edgar fetcher ~9s I/O bound (SEC EDGAR API latency) Reduce timeout; limit to most recent 1 filing; already parallel so doesn't block much
5 youtube fetcher ~3.6s I/O bound (transcript fetch) --no-youtube flag exists; transcript data has low density per token

7 Entity routing ~3.7s I/O bound (yfinance ticker check) Could cache known tickers; minor impact
8 API latency variance ±15s Uncontrollable Sonnet calls vary 8–21s depending on load. Accept.

Key insight: LLM output generation (not input processing) dominates. Cutting input tokens has minimal effect on wall time.

Section 6: Decision Cheat Sheet

Entity Routing

  • 這個 entity type 分類對嗎？NVIDIA → public_equity 正確嗎？（看 [pipeline] log 確認 ticker 解析）
  • 有沒有誤判的 edge case？例如 exchange 上市的 crypto 公司（Coinbase → 應該是 public_equity，不是 exchange）
  • DISAMBIGUATION 表裡有沒有缺漏的常見誤判？（Solana、TON 等已有，其他 L2/DeFi 有嗎？）
  • 有沒有新的 entity type 值得加？（例如 REIT、ETF、commodity）

Fetcher 環節

  • 這個 fetcher 對這個 entity type 有沒有產出有用的 data？（看 [strategy] active fetchers log + 看 extract input 裡那個 source 有多少內容）
  • 如果砍掉這個 fetcher，report 品質會變差嗎？（website 幾乎永遠 fast-fail → 可考慮對 public_equity 禁用）
  • 這個 fetcher 的耗時 vs 它提供的價值，ROI 合理嗎？（youtube 3.6s 但 transcript 只給 1500 chars → 邊際低）
  • sec_edgar 抓到的內容有多少進入最終 report？（截斷在 4000 chars，實際 filing 幾十頁）
  • rootdata 對非 Web3 private 公司（如 Anduril）覆蓋率接近 0 — 有更好的替代？

Extract 環節

  • Input 裡最肥的 source 是哪個？（input_chars log — 通常是 news 或 sec_edgar）
  • Output schema 裡哪些 field 對這個 entity type 是廢的？（已加 entity-type skip 指令，但有沒有還沒跳過的？）
  • Haiku 夠用嗎？目前看 NVIDIA/Solana 品質 OK；edge case 是資料量小但概念複雜的 private 公司
  • Temperature 0.2 合理嗎？同一間公司跑兩次，signals 順序/措辭應該一致
  • 4000 chars head-only truncation 對 news articles 是最糟的（尾部是最新文章）— 已知問題，head+tail 試過速度未改善

Judge 環節

  • 傳給 judge 的 facts 子集夠嗎？目前缺 valuation、competitive_landscape — judge 無法看到 P/E 或 peer multiples
  • Bull/bear case 有沒有提到最重要的 catalyst 和 risk？（看 phase_reasoning 是否引用了 facts 裡的具體數字）
  • Phase 判斷跟最近的 signals 一致嗎？如果 signals 全負面但 phase = EXPANDING，validate 應該要 flag
  • Macro risk 有沒有出現？（Rule 7 要求 bear_case 包含 macro/geopolitical risk — 對 NVIDIA 應提 export controls）
  • Missing data list 超過 50% → low_confidence: true → act 的 confidence 自動降為 medium/low

Act 環節

  • Entry/exit trigger 有具體數字嗎？（看 entry_trigger 欄位，不能只寫 "when revenue improves"）
  • Haiku 做 act 品質夠嗎？act 的 input 已包含 judgment 的 bull/bear/risks，Haiku 只是格式化 → 目前 OK
  • Wait_for 合理嗎？TIER A 優先用 upcoming_events；如果又寫了 "No confirmed catalyst" 需要問為什麼 upcoming_events 是空的
  • Guidance 和 segments 有沒有正確從 SEC/earnings 資料裡提取出來？

Validate 環節

  • Validate 有沒有抓到 judge/act 的邏輯錯誤？（看 narrative_gap 欄位 — 如果是 null 表示沒發現矛盾）
  • 有沒有放過明顯的矛盾？（Rule 9/10：completed events 不能出現在 upcoming_catalysts；entry price 不能高於說的 support level）
  • Validate 加的 13 秒值得嗎？它主要加了 stance、entry_trigger、risk_trigger、peer_context — 這四個欄位 act 無法可靠生成
  • 如果 validate 發現問題，現在是 patch（改寫 output）還是 reject + retry？→ 目前是 patch，沒有 retry 機制

Output / Formatter 環節

  • 最終 report 對用戶來說可讀嗎？關鍵資訊（verdict、stance、entry/risk trigger）在最後面，用戶要捲很久才看到
  • Structured JSON (Layer 1) 對 AI agent 夠用嗎？MCP consumer 能取得所有欄位？
  • so_what 欄位（"Watch: specific indicator"）有沒有真的可 actionable？

成本 & 速度

  • 這份 report 的 total cost 合理嗎？NVIDIA ≈ $0.055/run，目標 <$0.05 → 略超，主要是兩個 Sonnet calls（judge + validate）
  • 60-70 秒的等待用戶能接受嗎？MCP server 是 async，Claude 在等的時候可以做別的事
  • 哪個 stage 如果砍掉，省的時間 vs 失去的品質：
  • 砍 validate（-13s）：失去 stance、entry/risk trigger、peer_context、consistency check — 代價高
  • 合併 judge+act（-4s）：低風險，這兩步 input 差不多，合一個 Sonnet call 可省 1 次 API round-trip
  • 砍 youtube（-3.6s）：低代價，--no-youtube flag 已存在，transcript 資訊密度低
  • 砍 website（-0s）：幾乎免費，但目前就是 0s 所以沒差

What NOT to Optimize (已踩過的坑)

❌ Prompt Caching (cache_control: ephemeral)

試過: 在 get_llm_response() 的 Anthropic path 加 cache_control 標記 system prompt 結果: Run 1 vs Run 2 時間無差異（78s vs 84s） 原因: 瓶頸是 output generation，不是 input token 處理。cache 只省 input 處理時間，對生成速度無效。 結論: 不值得加複雜度。如果未來換成 extended thinking 或 batch API 另當別論。

❌ Input Token Pruning (prune_context / tiered truncation)

試過: 1. prune_context() — per-source 2000 token cap，head+tail truncation，總量 30k token 上限，tier-based source dropping 2. Tiered truncation in extract_facts — tier1=3000 chars, tier2=2000 chars, tier3=1500 chars

結果: 兩者都讓 extract 更慢或觸發 JSON parse error（max_tokens=4096 不夠裝完整 output） 原因: Extract input 從 40k chars 砍到 11k chars，但 extract 時間從 23s → 27s（更慢）。確認了 input size 對速度無效，output size 才是瓶頸。 結論: 不要動 input truncation。現有 4000 chars head-only truncation 夠用。

❌ CRITICAL rules block at prompt top

試過: 在 EXTRACT/JUDGE/ACT/VALIDATE system prompt 最前面加 6-line CRITICAL block 結果: System prompt 變大 → input tokens 增加 → 對速度無正面影響，且測試發現 extract 從 23s → 32s 結論: 重要規則留在 body 就好。頂部加 3 條以上的 block 會讓 prompt 過重。
